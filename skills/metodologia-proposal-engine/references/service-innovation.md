# Service Innovation Framework
# Use when INNOVATION MODE is active in Layer 2.

## Three innovation types

| Type | Definition | Risk | Approach |
|------|-----------|------|----------|
| A — High contextualiz. | Service exists; focus/sector/cases change | Low | Use canonical + deep brief |
| B — Variant | Same type, adjusted modules/duration/combination | Medium | Derive from canonical + mark deviations |
| C — New | Problem has no precedent in catalog | High | Compose from building blocks + [POR CONFIRMAR] all novel pricing |

**Type detection shortcut:** if you're adjusting WHAT the client does (cases, examples, industry) → Type A. If you're adjusting HOW the service runs (modules, sessions, duration) → Type B. If the client's problem genuinely doesn't fit any existing type → Type C.

---

## Allowed service types (no new types without JM sign-off)

| Type | Duration | Use when |
|------|----------|----------|
| Workshop | 2–4h | Basic capability install; single event |
| Bootcamp | 12–24h distributed | Full system build; practice + final project |
| Consultive Workshop | 6–12 × 2h sessions | Strategy co-design; deliverable per session |
| Program | 8–20 weeks | Deep transformation; mentoring + workshops + capstone |

**Budget mismatch protocol:**
- Doesn't cover Bootcamp → offer Workshop + mention 100% credit
- Doesn't cover Program → offer Consultive Workshop
- Doesn't cover any service → be direct: "Minimum cost to solve this problem is X. We can phase it or explore options." Never degrade quality to fit budget — offer a different service.

## Positioning innovation without over-promising

**Structure:**
1. Acknowledge why the standard catalog doesn't fit exactly
2. Name what is adapted and why that better solves the problem (not "different" — "more targeted")
3. Be explicit about limits (what can be guaranteed vs. what is subject to confirmation)
4. Frame confirmation as professional diligence, not uncertainty

**Anchor phrases for custom proposals:**
```
"We designed this from the principles of [base type], adapting [X] 
to specifically address [client's problem]."

"The quality standard is the same; the focus is different."

"Deliverables marked * are design proposals — confirmed in the alignment session before kickoff."

"The price is indicative; confirmed with the detailed scope before signing. No commitment until then."
```

## Validation: is an Innovation Mode proposal ready to deliver?

```
□ Base type chosen and justified
□ Gap documented (standard vs. custom)
□ Deliverables classified: standard / adapted / new*
□ Indicative price, not presented as confirmed
□ [POR CONFIRMAR] items listed explicitly
□ Innovation Mode disclaimer included
□ Legal verification (references/legal-guardrails.md) completed
□ No result promises without evidence wrapper
□ Both ES + EN versions drafted
```
